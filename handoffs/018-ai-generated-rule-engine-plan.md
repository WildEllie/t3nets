## The Problem

The current `RuleBasedRouter` uses hand-written regex patterns for skill detection and action routing. This works at $0/call but:
- Requires 20-40+ regex patterns per skill (manual maintenance)
- Doesn't scale as the skill catalog grows
- Patterns are global, not tenant-aware — can't optimize for a tenant's specific enabled skill combination
- "release status" is ambiguous between `sprint_status` and `release_notes` — cross-skill disambiguation is brittle
- The rule router isn't even wired into `router.py` yet (all traffic goes to Sonnet)

## The Solution

**AI generates the regex rules.** Keep the $0 in-memory regex routing, but have an AI service build the patterns automatically from skill metadata when tenants enable/disable skills.

### Architecture (2 tiers)

1. **Tier 1: AI-generated compiled regex** — $0, <1ms. Per-tenant rule sets stored in DynamoDB, loaded into memory. Generated by prompting Claude with skill metadata (names, descriptions, triggers, actions). AI understands the full skill combination and generates non-overlapping patterns.

2. **Tier 2: AI reasoning fallback** — $0.02-0.05. When rules don't match:
   - Enabled skill matches → execute skill
   - Disabled skill matches → "contact your admin to enable this"
   - No skill → freeform conversational response (AI answers general questions)
   - Unmatched requests logged as training data

### Key Innovation

The rule engine rebuilds per-tenant when skills change. If `release_notes` is disabled for a tenant, the `sprint_status` patterns relax to catch "release" queries. If both are enabled, the AI generates disambiguation patterns. This is impossible with static, hand-maintained regex.
